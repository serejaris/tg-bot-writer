Урок 0001: как вести агента через задачи

Один навык: не просить агента «сделай всё», а вести его через доказуемый цикл.

Зачем это тебе

В сессии с ботом агент не просто написал код. Он прошёл цепочку: issue → sub-issues → контракты → красные тесты → код → сервер → Telethon E2E → закрытие задач. Это и есть рабочий паттерн управления агентом.

Твоя новая команда агенту:

«Разбей это на GitHub sub-issues, сначала контракты, потом красные тесты, потом код, потом E2E. Перед закрытием каждой задачи сделай push и комментарий с итогом.»

Мини-модель

Если ты хочешь Проси так
Чтобы агент не потерял контекст «Зафиксируй цель в parent issue»
Чтобы работа была управляемой «Сделай sub-issues через GitHub Relationships»
Чтобы код не был случайным «Сначала опиши контракты»
Чтобы агент не закрыл задачу слишком рано «Покажи красные тесты, потом зелёные»
Чтобы результат был настоящим «Проверь E2E глазами пользователя»

Практика

Ответь себе без подсказки:

  1. Что должно появиться раньше: код или красный тест?
  2. Где должна быть связь между задачами: в тексте issue или в GitHub Relationships?
  3. Когда можно закрывать issue: до push или после push?
  4. Почему зелёные unit-тесты не заменяют Telethon E2E?
Проверить себя
  1. Сначала красный тест.
  2. В GitHub Relationships.
  3. После push.
  4. Unit-тесты проверяют куски кода, а E2E проверяет реальный пользовательский путь.
