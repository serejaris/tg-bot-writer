Гайд: AI-разработка через агента

Как вести AI-агента от большой идеи до доказанного результата: issue → sub-issues → контракты → красные тесты → код → E2E → push → комментарий → закрытие.

Главный вывод урока:

С агентом нельзя работать одной большой просьбой «сделай бота». Агенту нужен управляемый цикл, где каждый этап имеет артефакт и доказательство готовности.

1. Базовый цикл

Шаг Что просить у агента Что считать доказательством
1. Цель Создать parent issue с понятным результатом, ограничениями и критериями готовности. Issue открыт, результат описан так, что его можно проверить.
2. Подзадачи Разбить цель на sub-issues через GitHub Relationships. Связи видны в GitHub как настоящая иерархия, а не только как ручные ссылки в тексте.
3. Контракты Сначала описать входы, выходы, ошибки, данные, side effects и ограничения. По контрактам можно написать тесты до реализации.
4. Красные тесты Написать тесты и показать, что они падают по правильной причине. Падение связано с отсутствием функции, а не с опечаткой, импортом или неправильной настройкой теста.
5. Код Сделать минимальную реализацию до зелёных тестов. Тесты проходят локально и, если есть сервер, в серверной среде.
6. E2E Проверить реальный пользовательский сценарий. Есть факт внешнего результата: бот ответил, сохранил пост, опубликовал или выполнил нужное действие.
7. Фиксация Сделать commit и push, затем написать комментарий в issue. Коммит виден в GitHub issue, комментарий содержит «сделано / проверено / следующий шаг».
8. Закрытие Закрыть issue только после проверки фактического результата. Есть ссылка на коммит, результат, тесты и E2E-проверку.

Главное правило: если E2E нашёл баг, это не «мелочь после готовности», а новая sub-issue или fix-issue.

2. Что зафиксировал этот урок

Кейс урока — серверный Telegram-бот. Работа стала учебным примером не из-за самого бота, а из-за способа управления агентом.

Практика Что сработало Как повторять дальше
GitHub issue как центр управления Решения, прогресс и итоги не потерялись в чате. Каждая большая задача начинается с parent issue.
Sub-issues через Relationships Связи задач видны в интерфейсе GitHub. Не заменять иерархию задач ручным чеклистом в одном issue.
Контракты перед кодом docs/bot-contracts.md помог договориться, что бот принимает, возвращает и сохраняет. Перед feature-issue заводить spec-issue или отдельный контрактный раздел.
Красные тесты Тесты заранее зафиксировали ожидаемое поведение. Сначала test-issue, потом feature-issue, затем реализация до зелёных тестов.
Маленькие fix-issues Проблемы с GLM, JSON, channel id и Markdown не смешались в одну большую правку. Каждую внешнюю проблему оформлять отдельной задачей, если она требует кода или проверки.
Telethon E2E Реальный пользовательский сценарий нашёл то, что unit-тесты не поймали. Для интеграций с Telegram, оплатами, браузером, почтой и API всегда делать E2E глазами пользователя.
Push перед закрытием GitHub успел связать коммиты с задачей. Не писать финальный комментарий до push.

3. Лучшие практики AI-разработки с агентами

3.1. Не проси «сделай всё»

Большая просьба заставляет агента одновременно проектировать, писать код, тестировать, деплоить и объяснять. Это почти всегда снижает управляемость.

Лучше:

«Сначала создай parent issue, предложи sub-issues, укажи порядок, риски и критерии готовности. К коду пока не переходи.»

3.2. Контракт раньше кода

Контракт — это короткое соглашение о поведении системы. В нём должны быть входы, выходы, ошибки, внешние сервисы, формат данных и что нельзя делать.

Раздел контракта Вопрос агенту
Вход Что пользователь или внешний сервис отправляет системе?
Выход Что система должна вернуть, сохранить или опубликовать?
Ошибки Что делать, если модель, Telegram, JSON, сеть или файл дали неожиданный результат?
Side effects Какие действия меняют внешний мир: публикация, запись файла, отправка сообщения, изменение статуса?
Безопасность Какие секреты используются и где они не должны появиться?

3.3. Красный тест — защита от «почти готово»

Попроси агента показать сначала падающий тест. Потом попроси реализовать минимальную правку, чтобы тест стал зелёным. Так критерий готовности появляется до кода.

Промпт:

«Напиши тесты под контракт. Сначала запусти их и покажи, что они падают по ожидаемой причине. Реализацию пока не делай.»

3.4. Unit-тесты не заменяют E2E

Unit-тесты проверяют куски кода. E2E проверяет реальный путь пользователя через внешние сервисы. В этом уроке именно E2E показал проблемы с Telegram, GLM-ответом, JSON и Markdown.

Промпт:

«Проверь это не только тестами. Пройди реальный пользовательский сценарий: что отправляет пользователь, что отвечает бот, где сохраняется результат, что видно в логах.»

3.5. Внешние модели проверяются smoke-запросом

Если пайплайн зависит от LLM, сначала нужен минимальный запрос к модели. Не начинай отладку бизнес-логики, пока не доказано, что модель отвечает в нужном формате.

  • проверить, что ключ доступен в окружении;
  • сделать короткий smoke-запрос;
  • проверить, что max_tokens хватает для ответа;
  • обработать пустой content как ошибку, а не как нормальный результат;
  • быть готовым к ```json fenced JSON вместо голого JSON.

3.6. Секреты не попадают в git, issue и логи

Агент должен работать с секретами как с доступами, а не как с текстом для документации.

Можно фиксировать Нельзя фиксировать
GLM_API_KEY, TELEGRAM_BOT_TOKEN как имена переменных. Настоящие токены, session-файлы, приватные ключи.
.env.example с пустыми значениями. .env с реальными значениями.
Инструкцию, где ключ должен лежать. Ключ в README, PRD, issue, комментариях, логах, коммитах.

3.7. Серверная готовность тоже доказывается

Если код работает локально, это ещё не значит, что он работает на сервере.

  • проверить, что сервис активен;
  • проверить restart;
  • проверить последние логи;
  • проверить права на файлы с секретами;
  • проверить E2E уже в серверной среде.

3.8. Комментарий в issue — не отчёт ради отчёта

Комментарий должен помогать продолжить работу другому агенту или тебе через неделю.

Сделано:
- ...

Проверено:
- команда тестов;
- E2E-сценарий;
- серверный статус.

Коммит:
- ссылка или SHA.

Осталось:
- ...

Следующий шаг:
- ...

4. Definition of Done для agent-driven задачи

  • Parent issue содержит цель, ограничения и критерии готовности.
  • Большая задача разбита на sub-issues через Relationships.
  • Контракт написан до реализации.
  • Красные тесты были запущены и падали по правильной причине.
  • Реализация довела тесты до зелёного состояния.
  • Сделан E2E настоящим пользовательским сценарием.
  • Секреты не попали в git, issue, README, PRD, комментарии и логи.
  • Коммит запушен до финального комментария.
  • Issue закрыт только после фактической проверки результата.

5. Набор промптов

Parent issue

Создай parent issue для этой задачи.
Сначала зафиксируй цель, ограничения, критерии готовности, риски и что не входит в задачу.
К реализации не переходи.

Sub-issues

Разбей parent issue на sub-issues через GitHub Relationships.
Каждая sub-issue должна иметь один проверяемый результат.
Не делай один длинный чеклист вместо иерархии задач.

Контракт

Перед кодом опиши контракт: входы, выходы, ошибки, side effects, формат данных и секреты.
После этого объясни, какие тесты можно написать по этому контракту.

Красные тесты

Напиши тесты под контракт.
Запусти их до реализации и покажи, что они падают по ожидаемой причине.
Реализацию пока не делай.

E2E

Проверь результат настоящим пользовательским сценарием.
Опиши шаги: что отправил пользователь, что ответила система, где сохранился результат, какие логи подтверждают успех.

Перед закрытием

Перед закрытием issue проверь:
1. тесты зелёные;
2. E2E пройден;
3. секреты не попали в git и логи;
4. commit сделан и push выполнен;
5. комментарий в issue содержит сделано / проверено / коммит / следующий шаг.

6. Ошибки, которые теперь предотвращаем заранее

Ошибка Профилактика
Модель вернула пустой ответ. Smoke-запрос, достаточный max_tokens, явная обработка пустого content.
Модель вернула fenced JSON. Парсер должен принимать и голый JSON, и ```json.
Telegram channel id не сработал. Проверять числовой id через getChat и учитывать префикс -100 для каналов.
Markdown сломал служебный ответ. Служебные ответы отправлять без parse_mode, если форматирование не нужно.
Локально работает, на сервере нет. Проверять сервис, restart, логи и E2E на сервере.

Формула: агент пишет код быстро, но доверять нужно не скорости, а доказательствам: контрактам, тестам, E2E, логам и зафиксированным GitHub-комментариям.
