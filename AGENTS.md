# tg-bot-writer

Автоматизированное ведение Telegram-канала (контент-завод). Сейчас — только тексты и контент-план. Публикация по расписанию появится позже.

## Назначение и границы

- Этот репозиторий хранит **тексты постов** и **контент-план** для одного Telegram-канала.
- Источник правды о канале: `channel-brief.md` (аудитория, тон, темы, ограничения).
- Источник правды о постах: файлы в `posts/` и `content-plan/`.
- **Сейчас вне scope:** публикация в Telegram, cron, бот, деплой, правки кода автоматизации.
- Если пользователь просит «опубликовать» — напомни, что этап публикации ещё не подключён; сохрани пост как готовый черновик в `posts/scheduled/`.

## Канонические пути

| Путь | Назначение |
|------|------------|
| `channel-brief.md` | Профиль канала: для кого, о чём, как пишем |
| `content-plan/` | План тем: идеи, даты, статусы |
| `posts/drafts/` | Черновики по свежим идеям |
| `posts/scheduled/` | Готовые тексты с датой публикации |
| `posts/published/` | Архив уже вышедших постов (заполняется вручную или позже автоматически) |

## Главный сценарий: идея → пост → файл

Когда пользователь приходит с идеей («напиши пост про…», «оформи в пост…»):

1. **Прочитай** `channel-brief.md`. Если файла нет или в нём пустые секции — один раз уточни у пользователя аудиторию и тон, затем дополни brief.
2. **Уточни только критичное:** дата публикации, длина, CTA, ссылки, запретные темы. Не задавай больше 2–3 вопросов за раз.
3. **Напиши пост** на русском, в стиле канала из brief. Не выдумывай факты, цифры, цитаты и ссылки — если данных нет, спроси или пометь `[TODO: …]` в черновике.
4. **Сохрани файл** в `posts/drafts/` или `posts/scheduled/` (см. именование ниже).
5. **Обнови контент-план**, если пользователь назвал дату или тема идёт в план — запись в `content-plan/YYYY-MM.md` или отдельный файл идеи.
6. **Покажи пользователю** краткое превью поста и путь к сохранённому файлу.

## Формат файла поста

Каждый пост — Markdown с YAML frontmatter:

```yaml
---
title: "Короткий заголовок для файла"
date: 2026-06-18          # планируемая дата публикации или null
status: draft             # draft | scheduled | published
topic: "одна строка темы"
tags: [ai, tools]
source: "идея пользователя / контент-план / …"
created: 2026-06-18
---

Текст поста для Telegram.

Используй **жирный**, _курсив_, списки и эмодзи по правилам из channel-brief.md.
```

После frontmatter — **только текст поста**, готовый к копированию в Telegram. Без комментариев агента, без «вот вариант» и мета-объяснений.

## Именование файлов

- Черновик: `posts/drafts/YYYY-MM-DD-<slug>.md`
- С датой публикации: `posts/scheduled/YYYY-MM-DD-<slug>.md`
- `<slug>` — латиница, kebab-case, 2–5 слов по теме: `codex-opus-update`, `weekly-digest`

Если файл с таким именем уже есть — добавь суффикс `-2`, `-3`.

## Контент-план

Файл `content-plan/YYYY-MM.md` — таблица или список:

```md
| Дата | Тема | Статус | Файл |
|------|------|--------|------|
| 2026-06-20 | Обзор Codex | draft | posts/drafts/2026-06-20-codex-review.md |
```

Статусы: `idea` → `draft` → `scheduled` → `published`.

При новой идее без даты — добавь строку со статусом `idea`. При сохранении поста — обнови статус и колонку «Файл».

## Стиль и качество текста

- Язык постов — **русский**, если в `channel-brief.md` не указано иное.
- Обращение, длина, эмодзи, структура — строго из `channel-brief.md`.
- Telegram-разметка: `**bold**`, `_italic_`, ссылки `[текст](url)`. Не используй заголовки `#` в теле поста — они не нужны в Telegram.
- Один пост — одна мысль. Хук в первых 1–2 строках.
- Без воды, без «в современном мире», без канцелярита.
- Если пост длинный (>1500 символов) — предложи разбивку на серию или сокращение.

## Редактирование существующих постов

- Меняй только запрошенный файл; не переписывай соседние посты.
- Сохраняй frontmatter; обновляй `status` при смене стадии.
- Перед крупной правкой — прочитай 2–3 последних поста в `posts/` для一致ности тона.

## Git и коммиты

- Сообщения коммитов пиши **на русском языке**.
- Формат: краткий заголовок (до ~72 символов), при необходимости — пустая строка и 1–2 предложения с контекстом «зачем».
- Заголовок — в повелительном наклонении или от глагола: «Добавить…», «Обновить…», «Исправить…».
- Не дублируй в сообщении список всех изменённых файлов — опиши суть.
- Коммить и пушить только по явной просьбе пользователя.

## Безопасные действия по умолчанию

- Читать и писать файлы в `posts/`, `content-plan/`, `channel-brief.md`.
- Создавать новые посты и записи в контент-плане.
- Предлагать темы на неделю, если пользователь просит «контент-план» или «идеи на неделю».

## Требуют подтверждения

- Удаление файлов постов или строк контент-плана.
- Массовое переименование или перенос файлов между `drafts/` и `scheduled/`.
- Git commit и push.
- Любая публикация в Telegram или вызов внешних API.
- Изменение `channel-brief.md`, если пользователь не просил обновить профиль канала.

## Запрещено

- Публиковать посты в Telegram без явной команды пользователя.
- Хранить токены бота, API-ключи и секреты в репозитории.
- Выдумывать факты, статистику, цитаты, URL.
- Добавлять код автоматизации, CI, скрипты публикации — пока пользователь явно не перейдёт к следующему этапу.
- Создавать лишние README и документацию без запроса.

## Контракт выполнения

**Готово**, когда:

1. Пост сохранён в файл с корректным frontmatter и именем.
2. Контент-план обновлён (если применимо).
3. Пользователь видит превью и путь к файлу.

**Спроси**, если:

- Неясно, draft это или scheduled.
- Нет данных для фактических утверждений в посте.
- Идея противоречит ограничениям из `channel-brief.md`.

## Валидация перед завершением

- Frontmatter валиден, поле `status` заполнено.
- Тело поста не содержит `#`-заголовков и мета-комментариев агента.
- Имя файла соответствует дате и slug.
- Тон согласован с `channel-brief.md` и недавними постами.

## Следующий этап (не делать без команды)

Когда пользователь скажет перейти к автоматизации:

- Планировщик ежедневной публикации.
- Интеграция с Telegram Bot API.
- Перенос постов из `posts/scheduled/` в канал.

До этой команды оставайся в режиме **только тексты**.
